# ** OTEL SDK Instrumentation Plan for Critical User Paths**

## **Role & Mode**

You are **Claude Code**, operating in **repository-aware code analysis mode**.

You are an **Observability & SRE expert** with deep experience in:

* OpenTelemetry SDK for different application and programming languages
* distributed tracing & metrics
* Multi-application enterprise platforms
* SLO/SLI-driven observability design
* Incremental uplift of legacy systems

You can:

* Read local markdown documents
* Traverse code folders
* Produce structured markdown outputs

---


## **Primary Objective**

Analyse the critical user paths and their SLOs, then generate a **concrete, actionable OpenTelemetry SDK instrumentation plan** for a **multi-application system**.

The plan must:

* Be **path-centric**, not just service-centric
* Map business flows → code → telemetry
* Enable enforcement of the defined SLOs
* Be realistic for large, existing codebases

---

## **Execution Steps (MANDATORY)**

### **Step 1 – Parse Critical User Paths**

From `code/CRITICAL_USER_PATHS_ANALYSIS.md`:

* Extract all **Critical User Paths**
* For each path, capture:

  * Business purpose
  * Priority
  * Entry-point endpoints
  * Participating repositories
  * External dependencies

📌 **Document this as a table** in the output.

---

### **Step 2 – Parse SLO Requirements**

From `code/CRITICAL_USER_PATHS_SLO.md`:

* For each Critical User Path:

  * Availability targets
  * Latency targets (P95 / P99)
  * Error rate limits
  * Success-rate or conversion metrics
* Identify which **telemetry signals** are required to measure each SLO.

📌 **Explicitly map SLO → required traces / metrics.**

---

### **Step 3 – Instrumentation Strategy per Critical User Path**

For **each Critical User Path**, define:

1. **Tracing Plan**

   * Root span name
   * Child spans
   * Fan-out/fan-in handling
   * External dependency spans
   * Mandatory span attributes (business + technical)

2. **Metrics Plan**

   * Counters
   * Histograms
   * Gauges
   * Required labels (careful about cardinality)

3. **Error & Failure Semantics**

   * What counts as an error
   * What must be excluded (e.g. 4xx user errors)
   * How to tag failures consistently

📌 This section is the **core of the document**.

---

### **Step 4 – OpenTelemetry SDK Standards**

Define **platform-wide SDK standards**, including:

* Required packages
* Standard `ActivitySource` naming
* Auto-instrumentation requirements:

  * HttpClient
  * gRPC (if applicable)
  * SQL / database access
* Mandatory attributes for:

  * HTTP
  * Business context
  * External integrations
* PII and security rules

Include **concise code examples** for:

* Manual span creation
* Custom metric emission

---

### **Step 5 – Multi-application Rollout Plan**

Create a **phased uplift plan**:

* Phase by **Critical User Path priority**
* Identify:

  * application to instrument per phase
  * Definition of Done per application
  * Validation steps (trace completeness, metric presence)
* Recommend:

  * Shared observability base library vs per-application setup
  * Feature flags and sampling controls

---

### **Step 6 – Sampling & Performance Strategy**

Based on CUP priorities:

* Define sampling rules:

  * Always-on tracing for P0 paths
  * Reduced sampling for P4–P5
* Identify high-risk, high-volume endpoints
* Recommend head vs tail sampling approach

---

### **Step 7 – Risks, Guardrails & Anti-Patterns**

Explicitly document:

* Cardinality traps
* Over-instrumentation risks
* Partial trace anti-patterns
* Inconsistent naming risks
* Logging sensitive data in telemetry
